# Phase 3 卡住根因分析报告

- 分析日期：2026-03-07
- 分析对象：`phase3-advanced` 测试/联调阶段出现的卡住（hang）现象
- 输入依据：已知测试事实 + 仓库任务调度状态 + 监控日志行为

## 1. 现象定义

## 1.1 已确认现象

- Phase 3 模块可导入（导入耗时 0.20s）。
- `TestClient` 初始化可完成。
- `/health` 请求在部分环境/流程中出现超时。
- 并行可能同时启动多个服务（约 6 个）。

## 1.2 关键矛盾

- “导入成功 + 客户端可初始化” 与 “接口调用可能卡住” 同时存在。
- 说明阻塞更可能发生在：
  - 生命周期执行阶段（startup/shutdown）；
  - 请求处理阶段的资源等待；
  - 运行时竞争（端口/进程/锁）。

## 2. 根因假设评估

## 2.1 假设 A：并行启动多个 uvicorn 导致端口冲突或资源竞争

### 证据

- 多任务处于并行运行状态（`tasks/*.json` 多个 `running`）。
- 多个文档/脚本默认启动端口均为 `8000`（高冲突风险）。
- `phase3-advanced` 任务存在会话死亡与重启记录（`logs/monitor.log`）。

### 机制解释

- 端口冲突会导致某些服务启动失败或假启动，后续健康检查超时。
- 并发高负载下，I/O 与 CPU 竞争放大请求延迟，造成“看似卡住”。

### 置信度

- **高（首要根因）**

## 2.2 假设 B：TestClient 在特定环境下 lifespan 初始化阻塞

### 证据

- `TestClient` 构造成功，但 `/health` 请求阶段异常，符合 lifecycle 挂起特征。
- FastAPI/Starlette 在包含 startup 逻辑时，确有环境相关阻塞案例（尤其叠加并发或外部初始化）。

### 机制解释

- 测试客户端进入请求前需等待 lifespan 完成。
- 若 startup 内部存在慢初始化或等待依赖，首个请求会表现为超时。

### 置信度

- **中高（次要根因）**

## 2.3 假设 C：多进程导入 browser_use 触发遥测锁竞争

### 证据

- 已知 Phase 1 导入 1.17s 且伴随 `browser_use` 遥测行为；Phase 2/3 导入更快。
- 说明 `browser_use` 相关初始化可引入额外导入成本与不确定等待。

### 机制解释

- 多进程并发导入时，若遥测初始化涉及文件锁/网络/全局状态，可能产生竞争。
- 竞争通常表现为随机慢、偶发阻塞，而非稳定复现。

### 置信度

- **中（促发因素）**

## 3. 综合根因判断

Phase 3 卡住最可能不是单一故障，而是以下链式叠加：

1. **并发启动服务过多**（端口/资源竞争）→
2. **部分进程生命周期初始化变慢或异常**（TestClient 请求前等待）→
3. **browser_use 遥测在并发场景放大不确定性**（偶发锁竞争）

> 结论：A（主因）+ B（主通道）+ C（放大器）形成卡住。

## 4. 修复建议与落地优先级

## P0（立即执行）

1. **串行执行测试**：先确保单服务基线稳定，再恢复并行。
2. **端口硬隔离**：为每个 Phase/任务分配固定独立端口（例如 18001~18006）。
3. **禁用遥测**：测试环境统一设置 `BROWSERUSE_TELEMETRY=false`。

## P1（短期加固）

4. **关闭生命周期验证路径**：测试用例中增加 `lifespan=off` 变体，定位是否由 startup 阻塞触发。
5. **健康检查超时分层**：区分“服务未启动/端口被占用/请求阻塞”，输出明确错误码与日志标签。
6. **启动前端口探测**：启动脚本先检查端口占用，冲突即 fail-fast。

## P2（长期治理）

7. **测试编排器并发上限**：限制同机并发服务数（如 <=2）。
8. **引入稳定性回归用例**：新增 30~50 次 smoke 循环，统计 `/health` 超时率。
9. **观测埋点完善**：记录 startup 分段耗时（import、lifespan、first request）。

## 5. 建议验证矩阵（用于复现/回归）

- 维度 1：串行 vs 并行（1 / 2 / 6 服务）
- 维度 2：telemetry on vs off
- 维度 3：lifespan on vs off
- 维度 4：固定端口 vs 默认端口

判定标准：

- `/health` 成功率 >= 99%
- P95 首次请求耗时 < 1s
- 连续 30 次运行无 hang

## 6. 一句话结论

Phase 3 卡住的核心在于**并发运行环境不稳定**，而不是 Phase 3 代码不可用；先做并发与生命周期治理，可显著消除卡住。
